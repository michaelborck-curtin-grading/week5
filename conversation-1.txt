What are the most common accessibility problems in HTML contact forms?
Okay, so HTML contact forms can be a real pain for accessibility if they aren't built thoughtfully. Here are some of the most common issues that pop up and make it tough for everyone, especially users with disabilities:

Missing or Improper Labels The biggest culprit! Many forms use placeholder text inside the input field as a "label," but this disappears once someone starts typing, leaving them lost. Screen readers also often ignore or inconsistently announce placeholders. Every input needs a proper <label> element associated with it. 
Poor Keyboard Navigation and Focus Indicators If you can't tab through a form using only your keyboard, it's a huge barrier. Users need clear visual indicators (like outlines) to show which field is currently active so they know where they are in the form. 
Inaccessible Error Handling Just showing a red border for an error isn't enough, especially for users with visual impairments or color blindness. Error messages need to be clear, descriptive, tell users how to fix the problem, and be programmatically linked to the field so screen readers can announce them. 
Low Color Contrast Text, borders, and even placeholder text on forms often have insufficient color contrast, making them hard to read for users with low vision or certain color blindness. 
Complex or Inaccessible CAPTCHAs CAPTCHAs are meant to stop bots, but they often stop real people too, especially if they rely on deciphering distorted text or complex image recognition without accessible alternatives. 
Placeholder Text as Labels I mentioned this already, but it's worth reiterating how frequently this happens. Placeholders are for examples or hints, not substitutes for persistent labels. 
It's pretty wild how many forms out there still have these basic issues! Want to know how to spot these problems on a form?

Are there any tools I can use to help me assess accessibility?
Of course! There are a bunch of handy tools out there that can help you check for accessibility issues. Here are a few that are pretty popular and effective:

Browser Extensions (like Lighthouse in Chrome DevTools or axe DevTools): These are super easy to use right in your browser. Lighthouse is built into Chrome and can run a quick audit on your page, including accessibility. Axe DevTools is another fantastic extension that flags common issues and often suggests fixes, making it a great starting point for developers.
Automated Testing Tools (like WAVE or Siteimprove Accessibility Checker): These online tools allow you to enter a URL and get a report on potential accessibility problems. They can quickly scan for things like missing alt text, low contrast, and structural issues. They're great for getting a broad overview.
Screen Readers (like NVDA or JAWS): While not strictly "assessment" tools in the automated sense, using a screen reader is crucial for understanding how visually impaired users experience your forms. NVDA is free and open-source, and JAWS is a commercial option. It's the best way to really "feel" the user experience.
Manual Accessibility Checklists: No automated tool catches everything, so a manual checklist (like those based on WCAG guidelines) is essential. These checklists guide you through common pitfalls and specific criteria that need human judgment, like the clarity of error messages or the logical flow of content.
Using a combination of these will give you a pretty solid picture of your form's accessibility! Let me know if you want to dive deeper into any of them.

Which of the problems mention earleir would a WAVE audit catch and which would it miss?
That's a great question, and it really highlights the strengths and limitations of automated tools like WAVE!

Based on the problems we talked about earlier for HTML contact forms, here's what a WAVE audit would likely catch and what it would probably miss:

What a WAVE audit would likely catch:

Missing or Improper Labels: WAVE is excellent at detecting if input fields are missing associated <label> elements or if they're incorrectly linked (e.g., using for and id attributes). This is a pretty straightforward programmatic check.
Low Color Contrast: WAVE has a built-in contrast checker that can highlight text and background combinations that don't meet WCAG contrast ratios.
Placeholder Text as Sole Label: While it might not explicitly flag "placeholder as label," it would certainly flag the missing actual label, pushing you towards the correct solution.
Missing Language Declaration: Though not specifically a form issue, if your HTML page is missing a lang attribute, WAVE will definitely flag that.
What a WAVE audit would likely miss:

Poor Keyboard Navigation and Focus Indicators: Automated tools can tell if an element can be focused, but they can't effectively determine if the focus order is logical or if the visual focus indicator is clear and prominent enough for a human user. This requires manual testing with a keyboard.
Inaccessible Error Handling: While WAVE might flag if an error message isn't programmatically associated with its input field (e.g., using aria-describedby), it cannot assess if the error message itself is clear, descriptive, and actionable. It also can't tell if the error feedback is persistent or if it disappears too quickly. Human judgment is needed here.
Complex or Inaccessible CAPTCHAs: WAVE can't solve a CAPTCHA, nor can it determine if an alternative accessible method (like an audio CAPTCHA or a simple checkbox) is provided. It might flag some underlying structural issues, but the core accessibility problem of the CAPTCHA itself would be missed.
So, automated tools like WAVE are fantastic for catching the "low-hanging fruit" and many programmatic errors, but for things that require context, logical flow, or human interpretation, you'll still need manual testing!

What does a screen reader user actually experience when a form has no label
Oh man, when a form has no labels, it's pretty much a nightmare for a screen reader user. Imagine trying to fill out a paper form in a dark room with just someone pointing vaguely at spaces and saying "type here." That's kind of what it's like.

Here's what they'd typically experience:

They'd hear something generic: Instead of hearing "First Name" or "Email Address," the screen reader would announce something like "Edit text," "Text input," or "Unlabeled field." It gives them absolutely no context about what information is supposed to go into that box.
Confusion and Guesswork: They'd have to guess what each field is for, which is incredibly frustrating and often leads to errors. They might type their name into an email field or vice-versa.
No Help for Errors: If they do make a mistake, and the form has no proper error handling (which often goes hand-in-hand with missing labels), they might just get a generic error message like "Form submission failed" without knowing which field was incorrect or why.
Slow and Tedious Process: What takes a sighted user seconds becomes a slow, painful process of tabbing through fields, trying to infer meaning from surrounding text (if there is any), and hoping they're entering the right info. Many users would simply give up.
Essentially, a missing label disconnects the purpose of the input field from the field itself, making the form completely unusable for anyone relying on a screen reader. It's a huge barrier!

If I could only fix three thigns, which have the biggest impace
Okay, if you could only pick three things, hands down, these would be the ones to tackle for the biggest bang-for-your-buck impact:

Proper Labels for Every Input Field: Seriously, this is number one. It's the foundation of form accessibility. Without correct <label> elements linked to their inputs, screen reader users are completely lost. Fixing this immediately gives context and meaning to every field, making the form actually usable for them.
Robust Keyboard Navigation and Clear Focus Indicators: This impacts so many users – those who can't use a mouse, people with motor disabilities, and even power users who prefer keyboards. Ensuring a logical tab order and highly visible focus outlines makes the form navigable and understandable without a mouse. If you can't get into the fields, nothing else matters.
Accessible and Descriptive Error Handling: People make mistakes, and an inaccessible error message can be a dead end. Providing clear, specific, and programmatically announced error messages that explain what went wrong and how to fix it is crucial for successful form submission and prevents immense frustration. This directly impacts completion rates for users with disabilities.
Honestly, nailing these three will solve a huge percentage of the major accessibility blockers on most forms!
